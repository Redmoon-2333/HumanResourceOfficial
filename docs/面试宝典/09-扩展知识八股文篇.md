## 9.1 消息队列 (MQ)

### ❓ 消息队列有什么作用？

**参考答案**:

| 作用 | 说明 | 典型场景 |
|------|------|----------|
| **异步处理** | 将同步操作改为异步，提升响应速度 | 注册后发邮件/短信 |
| **削峰填谷** | 平滑流量峰值，保护下游系统 | 秒杀活动下单 |
| **服务解耦** | 降低系统间耦合度 | 订单→库存→物流 |
| **延时任务** | 精确控制任务执行时间 | 订单超时取消 |

### ❓ 消息队列如何保证消息不丢失？

**参考答案**:

**三步保证**:

```
1. 生产者不丢
   - 开启 confirm 确认机制
   - 消息持久化到磁盘
   - 失败重试机制

2. MQ 服务器不丢
   - 消息持久化 (交换机/队列/消息都持久化)
   - 镜像集群 (多副本)

3. 消费者不丢
   - 手动 ACK (处理成功后再确认)
   - 处理失败重回队列 (设置最大重试次数)
   - 记录日志，人工介入
```

### ❓ 消息队列如何保证消息顺序性？

**参考答案**:

**问题场景**:
```
订单创建 → 支付成功 → 发货完成
如果顺序错乱：先发货→再支付，业务出错
```

**解决方案**:

| 方案 | 实现 | 缺点 |
|------|------|------|
| **分区有序** | 相同 key 的消息发送到同一 partition | 只能保证局部有序 |
| **单队列串行** | 单个队列单消费者 | 吞吐量低 |
| **版本号控制** | 消息携带版本号，消费者按序处理 | 实现复杂 |

**RabbitMQ 实现**:
```java
// 将同一订单的消息发送到同一队列
String routingKey = "order." + orderId;  //  orderId 作为路由键
rabbitTemplate.convertAndSend("exchange", routingKey, message);
```

### ❓ 消息队列如何处理消息积压？

**参考答案**:

**排查步骤**:
```
1. 查看消费者状态 (是否宕机/卡顿)
2. 查看 MQ 监控 (消息生产/消费速率)
3. 查看日志 (是否有大量失败)
```

**解决方案**:

| 积压规模 | 方案 |
|----------|------|
| 少量 (万级) | 重启消费者，修复 bug |
| 中量 (十万级) | 增加消费者数量，提升消费能力 |
| 大量 (百万级) | 临时扩容：新建 Topic+ 批量转发+ 多倍消费者 |

**临时扩容方案**:
```
原队列：积压 100 万条消息，消费者处理能力 1000 条/秒
新方案:
  1. 新建 10 个临时队列
  2. 编写转发程序，将积压消息均匀分发到 10 个队列
  3. 启动 10 倍消费者，处理能力提升到 10000 条/秒
  4. 积压清理后，恢复原架构
```

### ❓ RabbitMQ/Kafka/RocketMQ 有什么区别？

**参考答案**:

| 特性 | RabbitMQ | Kafka | RocketMQ |
|------|----------|-------|----------|
| **吞吐量** | 万级 | 十万级 | 十万级 |
| **延迟** | 微秒级 | 毫秒级 | 毫秒级 |
| **可靠性** | 高 (ACK 机制) | 中 (可能丢消息) | 高 (支持事务) |

### ❓ 什么是死信队列？如何使用？

**参考答案**:

**死信 (Dead Letter) 来源**:

| 来源 | 说明 |
|------|------|
| **消息被拒绝** | basic.reject/nack 且 requeue=false |
| **消息过期** | TTL 超时 |
| **队列满** | 队列长度达到最大值 |

**死信队列作用**:

```
1. 异常消息隔离：失败消息单独存放，不影响正常消息
2. 问题排查：分析死信消息，定位业务问题
3. 人工处理：运维人员可以查看并手动处理
```

**RabbitMQ 配置示例**:

```java
// 普通队列配置死信
@Bean
public Queue businessQueue() {
    return QueueBuilder.durable("business.queue")
            .withArgument("x-message-ttl", 600000) // 10 分钟过期
            .withArgument("x-dead-letter-exchange", "dead.letter.exchange")
            .withArgument("x-dead-letter-routing-key", "dead")
            .build();
}

// 死信队列
@Bean
public Queue deadLetterQueue() {
    return QueueBuilder.durable("dead.letter.queue").build();
}
```

### ❓ RabbitMQ 如何保证消息可靠性？

**参考答案**:

**完整链路保证**:

```
┌─────────────────────────────────────────────────────────────┐
│                    MQ 可靠性保证                              │
├─────────────────────────────────────────────────────────────┤
│  1. 生产者不丢失                                              │
│     - 开启 confirm 确认模式                                   │
│     - 消息持久化到磁盘                                        │
│     - 失败重试 (最多 3 次)                                     │
│     - 本地消息表 (事务消息)                                    │
├─────────────────────────────────────────────────────────────┤
│  2. MQ 服务器不丢失                                           │
│     - Exchange 持久化                                          │
│     - Queue 持久化 (durable=true)                              │
│     - Message 持久化 (deliveryMode=2)                          │
│     - 镜像集群 (多副本)                                        │
├─────────────────────────────────────────────────────────────┤
│  3. 消费者不丢失                                              │
│     - 手动 ACK (处理成功后确认)                                │
│     - 失败重回队列 (限制重试次数)                              │
│     - 死信队列 (重试耗尽后接收)                                │
│     - 记录日志，人工介入                                      │
└─────────────────────────────────────────────────────────────┘
```

**RabbitMQ 实现代码**:

```java
// 生产者确认回调
rabbitTemplate.setConfirmCallback((correlationData, ack, cause) -> {
    if (!ack) {
        log.error("消息发送失败：{}", cause);
        // 记录到数据库，定时重试
    }
});

// 消费者手动 ACK
@RabbitListener(queues = "order.queue")
public void handleMessage(Message message, Channel channel, long deliveryTag) {
    try {
        // 处理业务
        processMessage(message);
        // 成功 ACK
        channel.basicAck(deliveryTag, false);
    } catch (Exception e) {
        try {
            // 失败重试 (false=不重新入队，进入死信)
            channel.basicNack(deliveryTag, false, false);
        } catch (IOException ex) {
            log.error("拒绝消息失败", ex);
        }
    }
}
```

### ❓ RabbitMQ 的 TTL 和延时队列如何实现？

**参考答案**:

**TTL (Time To Live)**:

| 设置方式 | 说明 | 代码 |
|---------|------|------|
| **队列级 TTL** | 队列中所有消息统一过期时间 | `x-message-ttl` |
| **消息级 TTL** | 每条消息独立的过期时间 | `expiration` 属性 |

**延时队列实现**:

```
方案 1: TTL + 死信队列 (推荐)
  消息 → 普通队列 (设置 TTL) → 过期后进入死信队列 → 消费者监听死信队列
  
方案 2: RabbitMQ 延时插件
  启用 rabbitmq_delayed_message_exchange 插件
  发送时设置 x-delay 属性
```

**代码示例**:

```java
// 方案 1: TTL + 死信
@Bean
public Queue delayQueue() {
    return QueueBuilder.durable("delay.queue")
            .withArgument("x-message-ttl", 60000) // 60 秒过期
            .withArgument("x-dead-letter-exchange", "dlx.exchange")
            .withArgument("x-dead-letter-routing-key", "dlx.routing")
            .build();
}

// 方案 2: 延时插件 (发送时)
rabbitTemplate.convertAndSend("exchange", "routingKey", message, msg -> {
    msg.getMessageProperties().setDelay(60000); // 60 秒延时
    return msg;
});
```

## 9.2 JVM (Java 虚拟机)

### ❓ JVM 内存区域是如何划分的？

**参考答案**:

```
┌─────────────────────────────────────────────────────────────┐
│                      JVM 内存结构                            │
├─────────────────────────────────────────────────────────────┤
│  线程私有:                                                   │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐         │
│  │  程序计数器  │  │  Java 虚拟机栈 │  │  本地方法栈  │         │
│  └─────────────┘  └─────────────┘  └─────────────┘         │
├─────────────────────────────────────────────────────────────┤
│  线程共享:                                                   │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                    堆 (Heap)                         │   │
│  │  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐ │   │
│  │  │ Eden 区      │  │ Survivor 0  │  │ Survivor 1  │ │   │
│  │  │ (8:1:1)     │  │  (From)     │  │  (To)       │ │   │
│  │  └─────────────┘  └─────────────┘  └─────────────┘ │   │
│  └─────────────────────────────────────────────────────┘   │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                  元空间 (Metaspace)                   │   │
│  │  (存储类信息、常量、静态变量)                           │   │

```

**各区域作用**:

| 区域 | 作用 | 常见问题 |
|------|------|----------|
| **程序计数器** | 记录当前执行位置 | 不会 OOM |
| **虚拟机栈** | 存储栈帧 (局部变量/操作数栈) | StackOverflowError |
| **本地方法栈** | Native 方法栈 | 少见 |
| **堆** | 存储对象实例 | OOM 高频区 |
| **元空间** | 存储类信息 (JDK8 替代永久代) | Metaspace OOM |

### ❓ 垃圾回收 (GC) 是如何工作的？

**参考答案**:

**GC 算法**:

| 算法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **标记 - 清除** | 标记可达对象→清除未标记 | 简单 | 内存碎片 |
| **标记 - 复制** | 存活对象复制到另一边 | 无碎片 | 空间利用率 50% |
| **标记 - 整理** | 存活对象向一端移动 | 无碎片 | 移动成本高 |

**分代回收理论**:
```
年轻代 (Young Generation):
  - Eden 区：新对象分配 (80%)
  - Survivor 区：存活的对象 (10% + 10%)
  - Minor GC：频繁，复制算法
  
老年代 (Old Generation):
  - 长期存活的对象
  - Major GC/Full GC：不频繁，标记 - 整理算法
```

**对象年龄判定**:
```
对象在 Survivor 区每熬过一次 Minor GC，年龄 +1
年龄达到阈值 (默认 15) → 晋升到老年代
```

### ❓ 常见的垃圾回收器有哪些？

**参考答案**:

| 垃圾回收器 | 区域 | 算法 | 特点 |
|-----------|------|------|------|
| **Serial** | 年轻代 | 复制 | 单线程，停顿时间长 |
| **ParNew** | 年轻代 | 复制 | Serial 多线程版 |
| **Parallel Scavenge** | 年轻代 | 复制 | 吞吐量优先 |
| **CMS** | 老年代 | 标记 - 清除 | 低延迟，首次并发 GC |
| **G1** | 全堆 | 分区回收 | 可预测停顿时间 |
| **ZGC** | 全堆 | 染色指针 | 停顿<10ms，JDK11+ |

**G1 收集器特点**:
```
- 将堆分为多个 Region
- 可预测停顿时间目标 (-XX:MaxGCPauseMillis)
- 避免全堆扫描，只回收价值高的 Region
- JDK9+ 默认收集器
```

### ❓ 什么是类加载机制？双亲委派模型是什么？

**参考答案**:

**类加载过程**:
```
加载 → 验证 → 准备 → 解析 → 初始化 → 使用 → 卸载
```

**双亲委派模型**:
```
                    ┌─────────────┐
                    │ Bootstrap   │
                    │ ClassLoader │
                    └──────┬──────┘
                           │ 委托
                    ┌──────▼──────┐
                    │  Extension  │
                    │ ClassLoader │
                    └──────┬──────┘
                           │ 委托
                    ┌──────▼──────┐
                    │ Application │
                    │ ClassLoader │
                    └──────┬──────┘
                           │ 委托
                    ┌──────▼──────┐
                    │  自定义     │
                    │ ClassLoader │
                    └─────────────┘
```

**工作原理**:
1. 类加载器收到加载请求
2. 先委托父加载器尝试加载
3. 父加载器无法加载时，子加载器才尝试加载

**优点**:
- 保证 Java 核心类的唯一性 (String 类不会被篡改)
- 避免类的重复加载
- 安全保护 (防止核心 API 被替换)

## 9.3 JUC (java.util.concurrent)

### ❓ volatile 关键字的作用是什么？

**参考答案**:

**两大特性**:

| 特性 | 说明 | 示例 |
|------|------|------|
| **可见性** | 一个线程修改后，其他线程立即可见 | 状态标志位 |
| **有序性** | 禁止指令重排序 | 双重检查锁定 |

**不保证原子性**:
```java
// ❌ volatile 不能保证原子性
private volatile int count = 0;
count++;  // 实际包含 读→改→写 三步，多线程不安全
```

**使用场景**:
```java
// ✅ 场景 1: 状态标志
private volatile boolean running = true;
public void stop() { running = false; }

// ✅ 场景 2: 单例模式双重检查
private static volatile Singleton instance;
public static Singleton getInstance() {
    if (instance == null) {
        synchronized (Singleton.class) {
            if (instance == null) {
                instance = new Singleton();
            }
        }
    }
    return instance;
}
```

### ❓ synchronized 和 ReentrantLock 有什么区别？

**参考答案**:

| 对比项 | synchronized | ReentrantLock |
|--------|--------------|---------------|
| **实现层面** | JVM 层面 (关键字) | API 层面 (类) |
| **释放锁** | 自动释放 | 手动 unlock() |
| **等待可中断** | ❌ 不可中断 | ✅ lockInterruptibly() |
| **公平锁** | ❌ 只能非公平 | ✅ 可构造公平锁 |
| **多条件等待** | ❌ 单一 wait/notify | ✅ 多个 Condition |
| **性能** | JDK6+ 优化后相近 | 略优 |

**代码对比**:
```java
// synchronized
synchronized (obj) {
    // 临界区
}

// ReentrantLock
ReentrantLock lock = new ReentrantLock();
lock.lock();
try {
    // 临界区
} finally {
    lock.unlock();  // 必须 finally 中释放
}
```

### ❓ 线程池的核心参数有哪些？

**参考答案**:

```java
new ThreadPoolExecutor(
    int corePoolSize,        // 核心线程数
    int maximumPoolSize,     // 最大线程数
    long keepAliveTime,      // 非核心线程空闲存活时间
    TimeUnit unit,           // 时间单位
    BlockingQueue<Runnable> workQueue,  // 任务队列
    ThreadFactory threadFactory,        // 线程工厂
    RejectedExecutionHandler handler    // 拒绝策略
)
```

**执行流程**:
```
新任务到达
    ↓
核心线程数未满 → 创建核心线程执行
    ↓
核心线程已满 → 放入队列
    ↓
队列已满 → 创建非核心线程
    ↓
达到最大线程数 → 触发拒绝策略
```

**拒绝策略**:

| 策略 | 行为 | 适用场景 |
|------|------|----------|
| **AbortPolicy** (默认) | 抛出 RejectedExecutionException | 不允许丢失 |
| **CallerRunsPolicy** | 调用者线程执行 | 降低提交速度 |
| **DiscardPolicy** | 直接丢弃 | 允许丢失 |
| **DiscardOldestPolicy** | 丢弃最老任务，重试 | 旧任务可丢弃 |

**参数配置建议**:

| 任务类型 | 核心线程数 | 最大线程数 | 队列 |
|---------|-----------|-----------|------|
| **CPU 密集型** | CPU 核数 | CPU 核数 +1 | 有界队列 |
| **IO 密集型** | CPU 核数 × 2 | CPU 核数 × 4 | 有界队列 |
| **混合型** | 根据测试调整 | 根据测试调整 | 有界队列 |

### ❓ ThreadLocal 是什么？有什么使用场景和问题？

**参考答案**:

**原理**:
```
ThreadLocal 为每个线程提供独立的变量副本
每个线程访问 ThreadLocal 时，实际访问的是自己线程内部的副本
```

**使用场景**:
```java
// 场景 1: 数据库连接
private static final ThreadLocal<Connection> connectionHolder = 
    ThreadLocal.withInitial(() -> createConnection());

// 场景 2: 用户上下文 (Spring Security 的 SecurityContext)
private static final ThreadLocal<UserContext> userContextHolder = 
    new ThreadLocal<>();

// 场景 3: SimpleDateFormat 线程安全
private static final ThreadLocal<SimpleDateFormat> dateFormatHolder = 
    ThreadLocal.withInitial(() -> new SimpleDateFormat("yyyy-MM-dd"));
```

**内存泄漏问题**:
```
ThreadLocalMap 的 Entry 继承自 WeakReference
Key 是弱引用 (指向 ThreadLocal)，Value 是强引用

问题：如果 ThreadLocal 被 GC 回收，Key 变为 null，但 Value 仍被强引用
解决：使用完后必须调用 remove()
```

**正确使用方式**:
```java
try {
    dateFormatHolder.get().parse(dateString);
} finally {
    dateFormatHolder.remove();  // 必须清理
}
```

### ❓ CountDownLatch/CyclicBarrier/Semaphore 有什么区别？

**参考答案**:

| 工具类 | 作用 | 可复用 | 典型场景 |
|--------|------|--------|----------|
| **CountDownLatch** | 一个或多个线程等待其他线程完成 | ❌ 一次性 | 组装车 (等所有零件安装完) |
| **CyclicBarrier** | 一组线程互相等待，都到达屏障后继续 | ✅ 可重用 | 多玩家游戏 (等人齐后开始) |
| **Semaphore** | 控制同时访问的线程数 (信号量) | ✅ 可重用 | 数据库连接池 (限流) |

**代码示例**:
```java
// CountDownLatch: 等待 3 个任务完成
CountDownLatch latch = new CountDownLatch(3);
for (int i = 0; i < 3; i++) {
    new Thread(() -> {
        doWork();
        latch.countDown();
    }).start();
}
latch.await();  // 等待 3 个任务完成
System.out.println("所有任务完成");

// CyclicBarrier: 3 个线程互相等待
CyclicBarrier barrier = new CyclicBarrier(3, () -> {
    System.out.println("所有人到达，开始游戏");
});
// 每个线程调用 barrier.await() 等待其他人

// Semaphore: 限制同时访问的线程数
Semaphore semaphore = new Semaphore(5);  // 最多 5 个并发
semaphore.acquire();  // 获取许可
try {
    doWork();
} finally {
    semaphore.release();  // 释放许可
}
```

## 9.4 操作系统

### ❓ 进程和线程有什么区别？

**参考答案**:

| 对比项 | 进程 | 线程 |
|--------|------|------|
| **定义** | 资源分配的最小单位 | CPU 调度的最小单位 |
| **内存空间** | 独立地址空间 | 共享进程内存 |
| **通信方式** | IPC(管道/消息队列/共享内存) | 直接读写共享变量 |
| **创建开销** | 大 (分配资源) | 小 (共享资源) |
| **切换开销** | 大 (切换页表) | 小 (只切换栈和寄存器) |
| **健壮性** | 一个进程崩溃不影响其他 | 一个线程崩溃可能影响整个进程 |

**类比**:
```
进程 = 工厂
线程 = 工厂里的工人

工厂之间独立 (进程独立)
工人共享工厂资源 (线程共享内存)
```

### ❓ 什么是死锁？如何预防和避免？

**参考答案**:

**死锁四个必要条件**:
```
1. 互斥条件：资源只能被一个线程占用
2. 请求与保持：持有资源的同时请求其他资源
3. 不剥夺：已获得的资源不能被强制剥夺
4. 循环等待：存在循环等待链 A→B→C→A
```

**死锁示例**:
```java
// 线程 1
synchronized (obj1) {
    synchronized (obj2) {
        // ...
    }
}

// 线程 2
synchronized (obj2) {
    synchronized (obj1) {
        // ...
    }
}
```

**预防和避免**:

| 方法 | 原理 | 实现 |
|------|------|------|
| **破坏请求与保持** | 一次性申请所有资源 | 预分配 |
| **破坏不剥夺** | 资源可被强制剥夺 | 超时机制 |
| **破坏循环等待** | 按顺序申请资源 | 有序编号 |
| **避免死锁** | 银行家算法 | 预判是否安全 |

**实用建议**:
```
1. 固定加锁顺序 (总是先 A 后 B)
2. 使用定时锁 (tryLock 超时)
3. 使用高级工具 (ReentrantLock, Semaphore)
4. 减少锁粒度 (细粒度锁)
```

### ❓ 操作系统有哪些调度算法？

**参考答案**:

| 算法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **FCFS** | 先来先服务 | 简单 | 短任务等待时间长 |
| **SJF** | 短作业优先 | 平均等待时间最短 | 长任务可能饿死 |
| **优先级调度** | 优先级高的先执行 | 灵活 | 低优先级可能饿死 |
| **时间片轮转** | 每个任务执行一个时间片 | 公平，响应快 | 频繁切换开销大 |
| **多级反馈队列** | 多个优先级队列，可升降级 | 综合最优 | 实现复杂 |

**Linux 调度器**:
```
CFS (Completely Fair Scheduler):
- 维护红黑树，按 vruntime 排序
- 选择 vruntime 最小的任务执行
- 保证每个任务获得公平的 CPU 时间
```

## 9.5 计算机网络

### ❓ TCP 三次握手和四次挥手的原理？

**参考答案**:

**三次握手** (建立连接):
```
Client                    Server
   │                         │
   │──── SYN (seq=x) ───────▶│  第一次：Client 发起连接
   │                         │
   │◀─── SYN+ACK (seq=y,ack=x+1) ──│  第二次：Server 确认
   │                         │
   │──── ACK (ack=y+1) ─────▶│  第三次：Client 确认
   │                         │
   │===== 连接已建立 ======    │
```

**为什么三次**:
- 防止已失效的连接请求突然到达
- 确认双方的发送和接收能力
- 同步初始序列号

**四次挥手** (断开连接):
```
Client                    Server
   │                         │
   │──── FIN (seq=u) ──────▶│  第一次：Client 请求关闭
   │                         │
   │◀──── ACK (ack=u+1) ─────│  第二次：Server 确认
   │                         │
   │===== Client→Server 关闭 ======│
   │                         │
   │◀──── FIN (seq=v) ───────│  第三次：Server 请求关闭
   │                         │
   │──── ACK (ack=v+1) ─────▶│  第四次：Client 确认
   │                         │
   │===== 连接已断开 ======    │
```

**为什么四次**:
- TCP 是全双工的，两个方向的关闭需要独立进行
- Server 收到 FIN 后，可能还有数据要发送

### ❓ TCP 如何保证可靠传输？

**参考答案**:

| 机制 | 作用 | 实现方式 |
|------|------|----------|
| **校验和** | 检测数据损坏 | 发送方计算，接收方验证 |
| **序列号** | 解决乱序问题 | 每个字节编号，按序重组 |
| **确认应答** | 确保数据到达 | ACK 确认，超时重传 |
| **超时重传** | 处理丢包 | 超时未收到 ACK 则重传 |
| **流量控制** | 防止接收方缓冲区溢出 | 滑动窗口机制 |
| **拥塞控制** | 防止网络拥塞 | 慢启动/拥塞避免/快重传/快恢复 |

**滑动窗口**:
```
发送窗口内的数据可以连续发送
收到 ACK 后，窗口向前滑动
窗口大小 = min(接收窗口，拥塞窗口)
```

### ❓ HTTP 和 HTTPS 有什么区别？

**参考答案**:

| 对比项 | HTTP | HTTPS |
|--------|------|-------|
| **协议** | 应用层 (基于 TCP) | HTTP + SSL/TLS |
| **端口** | 80 | 443 |
| **加密** | ❌ 明文传输 | ✅ 加密传输 |
| **证书** | ❌ 无需证书 | ✅ 需要 CA 证书 |
| **性能** | 快 (无加密开销) | 略慢 (加密/解密) |
| **安全性** | 低 (中间人攻击) | 高 (防窃听/篡改) |

**HTTPS 加密过程**:
```
1. Client 发送 ClientHello (支持的加密套件)
2. Server 返回 ServerHello + 证书 + 公钥
3. Client 验证证书 → 生成随机数 → 用公钥加密发送
4. Server 用私钥解密 → 获得随机数
5. 双方使用随机数生成对称密钥
6. 后续通信使用对称加密 (AES)
```

### ❓ GET 和 POST 有什么区别？

**参考答案**:

| 对比项 | GET | POST |
|--------|-----|------|
| **用途** | 查询数据 | 提交数据 |
| **幂等性** | ✅ 幂等 | ❌ 不幂等 |
| **参数位置** | URL 中 | 请求体中 |
| **参数长度** | 有限制 (~2KB) | 无限制 |
| **安全性** | 低 (参数暴露在 URL) | 高 (参数在 body) |
| **缓存** | 可缓存 | 不缓存 |
| **编码** | 只能 URL 编码 | 支持多种编码 |

**语义区别**:
```
GET:  从服务器获取资源 (不应有副作用)
POST: 向服务器提交数据 (可能有副作用)
PUT:  全量更新资源
PATCH: 部分更新资源
DELETE: 删除资源
```

## 9.6 数据库进阶

### ❓ 数据库事务的 ACID 特性是什么？

**参考答案**:

| 特性 | 含义 | 实现机制 |
|------|------|----------|
| **原子性 (Atomicity)** | 事务要么全部成功，要么全部失败 | Undo Log |
| **一致性 (Consistency)** | 事务前后数据状态一致 | 约束/触发器 |
| **隔离性 (Isolation)** | 事务之间互不干扰 | 锁/MVCC |
| **持久性 (Durability)** | 事务提交后永久保存 | Redo Log |

### ❓ 数据库隔离级别有哪些？

**参考答案**:

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | MySQL 默认 |
|---------|------|-----------|------|-----------|
| **读未提交** | ✅ 可能 | ✅ 可能 | ✅ 可能 | ❌ |
| **读已提交** | ❌ 避免 | ✅ 可能 | ✅ 可能 | Oracle/PG 默认 |
| **可重复读** | ❌ 避免 | ❌ 避免 | ⚠️ 可能 (MVCC 缓解) | ✅ MySQL 默认 |
| **串行化** | ❌ 避免 | ❌ 避免 | ❌ 避免 | ❌ |

**并发读问题**:
```
脏读：读到其他事务未提交的数据
不可重复读：同一事务内两次读取结果不同
幻读：同一事务内两次查询，第二次多了/少了数据
```

### ❓ MySQL 的索引数据结构是什么？

**参考答案**:

**B+ 树特点**:
```
1. 非叶子节点只存索引，不存数据 (单节点存更多索引)
2. 叶子节点存所有数据 (数据页)
3. 叶子节点用链表连接 (范围查询高效)
4. 树高度通常 3-4 层 (10 亿数据)
```

**B+ 树 vs B 树**:
| 对比项 | B 树 | B+ 树 |
|--------|-----|-------|
| 数据存储位置 | 所有节点 | 只叶子节点 |
| 叶子节点链接 | ❌ 无 | ✅ 有链表 |
| 范围查询 | 慢 (需要中序遍历) | 快 (链表遍历) |
| MySQL 使用 | ❌ | ✅ |

**聚簇索引 vs 非聚簇索引**:
```
聚簇索引 (主键索引):
  叶子节点 = 数据行
  一张表只能有一个

非聚簇索引 (二级索引):
  叶子节点 = 主键值
  需要回表查询数据
  一张表可以有多个
```

### ❓ 什么是 MVCC？如何实现？

**参考答案**:

**MVCC (多版本并发控制)**:
```
核心思想：数据有多个版本，读操作读快照，写操作创建新版本
解决的问题：读写不阻塞，提高并发性能
```

**实现原理**:
```
1. 隐藏字段
   - DB_TRX_ID: 最后修改的事务 ID
   - DB_ROLL_PTR: 回滚指针 (指向上一个版本)

2. Undo Log
   记录数据的历史版本链

3. Read View (读视图)
   快照时刻，包含活跃事务列表
```

**版本链查找规则**:
```
当前版本 → Undo Log → 上一版本 → ... → 最早版本

查找时：
- 事务 ID < Read View 最小活跃 ID → 可见
- 事务 ID 在活跃事务列表中 → 不可见
- 事务 ID > Read View 最大活跃 ID → 不可见
```

**MVCC + 锁 解决幻读**:
```
快照读 (普通 SELECT): MVCC
当前读 (SELECT ... FOR UPDATE): Next-Key Lock
```

## 9.7 Redis 进阶

### ❓ Redis 为什么这么快？

**参考答案**:

| 因素 | 说明 |
|------|------|
| **纯内存操作** | 数据在内存中，访问速度快 (纳秒级) |
| **单线程模型** | 无线程切换/竞争开销 (命令串行执行) |
| **IO 多路复用** | epoll/kqueue，单线程处理多个连接 |
| **高效数据结构** | SDS/哈希表/跳表/压缩列表 |
| **合理过期策略** | 惰性删除 + 定期删除 |

### ❓ Redis 有哪些数据类型？使用场景？

**参考答案**:

| 数据类型 | 结构 | 典型场景 |
|---------|------|----------|
| **String** | 动态字符串 | 缓存/计数器/分布式锁 |
| **List** | 双向链表 | 消息队列/最新列表 |
| **Hash** | 哈希表 | 对象存储/购物车 |
| **Set** | 哈希表 (去重) | 标签/共同好友 |
| **ZSet** | 跳表 + 哈希表 | 排行榜/优先级队列 |
| **Bitmap** | 位数组 | 签到/布隆过滤器 |
| **HyperLogLog** | 概率算法 | UV 统计 (允许误差) |
| **Geo** | GeoHash | 地理位置/附近的人 |

### ❓ Redis 持久化方式有哪些？

**参考答案**:

| 方式 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|----------|
| **RDB** | 定时快照 | 文件小，恢复快 | 可能丢失数据 | 容忍少量丢失 |
| **AOF** | 记录写操作 | 数据更安全 | 文件大，恢复慢 | 高一致性要求 |
| **混合** | RDB+AOF | 两者优点 | 配置复杂 | 生产推荐 |

**RDB 触发条件**:
```
save 900 1      # 900 秒内至少 1 个 key 变化
save 300 10     # 300 秒内至少 10 个 key 变化
save 60 10000   # 60 秒内至少 10000 个 key 变化
```

**AOF 重写机制**:
```
当 AOF 文件过大时，后台进程重写：
- 合并多条写操作 (set a 1; set a 2 → set a 2)
- 不读旧 AOF，直接读当前内存快照
```

### ❓ Redis 主从复制原理？

**参考答案**:

**复制流程**:
```
1. Slave 启动，向 Master 发送 SYNC 命令
2. Master 执行 BGSAVE 生成 RDB 快照
3. Master 将 RDB 发送给 Slave
4. Slave 加载 RDB 恢复数据
5. Master 将缓冲区的写命令发送给 Slave
6. 后续写操作实时同步给 Slave
```

**增量复制** (PSYNC):
```
Slave 断线重连后：
- 发送已复制的 offset
- Master 从 offset 处继续发送
- 无需全量复制
```
