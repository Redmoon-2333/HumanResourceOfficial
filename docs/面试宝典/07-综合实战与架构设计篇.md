### 📁 本项目中的体现

**Controller 层** (AuthController.java):

```java
@RestController
@RequestMapping("/api/auth")
public class AuthController {
    
    @Autowired
    private AuthService authService;
    
    @PostMapping("/login")
    @RateLimit(key = "'auth:login:' + #request.username", maxRequests = 5, windowSize = 60)
    public ApiResponse<Map<String, Object>> login(@RequestBody @Validated LoginRequest request) {
        // 1. 参数校验 (@Validated)
        // 2. 调用 Service 层
        Map<String, Object> result = authService.login(request);
        // 3. 统一响应封装
        return ApiResponse.success("登录成功", result);
    }
}
```

**Service 层** (ActivityService.java):

```java
@Service
public class ActivityService {
    
    @Autowired
    private ActivityMapper activityMapper;
    
    @DistributedLock(key = "'activity:create:' + #activity.activityName")
    @CacheEvict(value = "activity:list", key = "'all'")
    public Activity createActivity(Activity activity) {
        // 1. 业务逻辑
        activityMapper.insert(activity);
        // 2. 缓存清除 (@CacheEvict 后置执行)
        // 3. 分布式锁释放 (切面自动处理)
        return activity;
    }
}
```

## 7.2 微服务拆分思路

### ❓ 面试官问：如果要将这个项目改造成微服务，你会如何拆分？

### ✅ 参考回答

**微服务拆分原则**:

```
1. 单一职责：每个服务只负责一个业务域
2. 高内聚低耦合：相关业务放在一起
3. 数据独立性：每个服务有自己的数据库
4. 可独立部署：服务间松耦合
```

**本项目拆分方案**:

```
┌─────────────────────────────────────────────────────────────┐
│                    微服务拆分架构                            │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐      │
│  │  用户服务    │  │  活动服务    │  │  AI 服务       │      │
│  │  User Svc    │  │ Activity Svc │  │  AI Chat Svc │      │
│  ├──────────────┤  ├──────────────┤  ├──────────────┤      │
│  │ - 用户管理   │  │ - 活动 CRUD   │  │ - 智能对话   │      │
│  │ - 激活码    │  │ - 图片管理   │  │ - RAG 检索   │      │
│  │ - 权限角色   │  │ - 物资管理   │  │ - 策划生成   │      │
│  └──────────────┘  └──────────────┘  └──────────────┘      │
│        ↓                  ↓                  ↓               │
│     user_db          activity_db         vector_db          │
│                                                              │
└─────────────────────────────────────────────────────────────┘
                            ↑
                    ┌───────────────┐
                    │  API Gateway  │
                    │  - 统一鉴权   │
                    │  - 路由转发   │
                    │  - 限流熔断   │
                    └───────────────┘
```

**服务边界**:

| 服务 | 包含模块 | 数据库 | 端口 |
|------|---------|--------|------|
| **user-service** | 用户管理、激活码、角色权限 | user_db | 8081 |
| **activity-service** | 活动管理、物资管理、图片管理 | activity_db | 8082 |
| **ai-chat-service** | AI 对话、RAG 知识库、策划生成 | vector_db | 8083 |

**服务间通信**:

```
同步调用 (REST):
  activity-service → user-service (验证用户权限)
  
异步调用 (MQ):
  user-service --[用户注册事件]--> activity-service
  activity-service --[活动创建事件]--> ai-chat-service
```

## 7.3 分布式事务方案

### ❓ 面试官问：微服务架构下如何保证分布式事务一致性？

### ✅ 参考回答

**分布式事务解决方案**:

| 方案 | 适用场景 | 优点 | 缺点 |
|------|---------|------|------|
| **2PC/XA** | 强一致性要求 | 数据强一致 | 性能差，单点故障 |
| **TCC** | 核心支付场景 | 最终一致，性能好 | 实现复杂 |
| **Saga** | 长流程业务 | 简单易实现 | 补偿逻辑复杂 |
| **本地消息表** | 异步场景 | 简单可靠 | 有延迟 |
| **MQ 事务** | 解耦场景 | 高吞吐 | 实现复杂 |

**本项目推荐方案**:

```
场景 1: 用户注册 + 发放激活码
  → 本地事务 (同一数据库)

场景 2: 活动创建 + 通知 AI 学习
  → MQ 事务消息 (最终一致性)

场景 3: 跨服务调用 (活动报名 + 扣减物资)
  → TCC 模式 (Try-Confirm-Cancel)
```

**Saga 模式示例**:

```
活动报名流程:

正向操作:
  1. 创建活动 (Activity Svc)
  2. 扣减物资 (Material Svc)
  3. 发送通知 (Notification Svc)

补偿操作 (任一失败时):
  1. 取消活动
  2. 恢复物资
  3. 发送取消通知
```

## 7.4 消息队列应用

### ❓ 面试官问：消息队列在项目中有什么应用场景？

### ✅ 参考回答

**MQ 核心应用场景**:

| 场景 | 说明 | MQ 优势 |
|------|------|---------|
| **异步处理** | 注册后发邮件/短信 | 提升响应速度 |
| **削峰填谷** | 秒杀活动下单 | 平滑流量峰值 |
| **服务解耦** | 订单→库存→物流 | 降低耦合度 |
| **延时任务** | 订单超时取消 | 精确延时控制 |

**技术选型对比**:

| MQ | 吞吐量 | 延迟 | 可靠性 | 适用场景 |
|----|--------|------|--------|----------|
| **RabbitMQ** | 中 | 低 | 高 | 中小型系统 |
| **Kafka** | 极高 | 中 | 中 | 日志收集/大数据 |
| **RocketMQ** | 高 | 低 | 极高 | 金融/电商核心 |
| **Redis Stream** | 中 | 极低 | 中 | 简单异步场景 |

**本项目引入 MQ 改造**:

```java
// 活动创建后异步通知 AI 学习
@Service
public class ActivityService {
    
    @Autowired
    private RabbitTemplate rabbitTemplate;
    
    @Transactional
    public Activity createActivity(Activity activity) {
        // 1. 创建活动
        activityMapper.insert(activity);
        
        // 2. 发送 MQ 消息 (异步通知 AI 服务)
        rabbitTemplate.convertAndSend(
            "activity.exchange",
            "activity.created",
            activity
        );
        
        return activity;
    }
}

// AI 服务监听
@Component
public class AIChatListener {
    
    @RabbitListener(queues = "ai.activity.queue")
    public void handleActivityCreated(Activity activity) {
        // 异步学习活动内容，丰富知识库
        ragService.addKnowledge(activity);
    }
}
```

### 📁 本项目中的 MQ 实现

**配置类** (RabbitMQConfig.java):

```java
@Configuration
public class RabbitMQConfig {

    // Topic 交换机
    @Bean
    public TopicExchange activityExchange() {
        return new TopicExchange("activity.exchange", true, false);
    }
    
    // 队列配置 (带 TTL 和死信)
    @Bean
    public Queue activityCreatedQueue() {
        return QueueBuilder.durable("activity.created.queue")
                .withArgument("x-message-ttl", 600000) // 10 分钟过期
                .withArgument("x-dead-letter-exchange", "dead.letter.exchange")
                .withArgument("x-dead-letter-routing-key", "dead")
                .build();
    }
    
    // 绑定关系
    @Bean
    public Binding activityCreatedBinding(Queue activityCreatedQueue, 
                                           TopicExchange activityExchange) {
        return BindingBuilder.bind(activityCreatedQueue)
                .to(activityExchange)
                .with("activity.created");
    }
    
    // JSON 消息转换器
    @Bean
    public MessageConverter jsonMessageConverter() {
        return new Jackson2JsonMessageConverter();
    }
}
```

**Service 发送消息** (ActivityService.java):

```java
@Service
@RequiredArgsConstructor
public class ActivityService {

    private final MQSender mqSender;
    
    @DistributedLock(key = "'activity:create:' + #activity.activityName")
    public Activity createActivity(Activity activity) {
        // 1. 数据库操作
        activityMapper.insert(activity);
        
        // 2. 清除缓存
        // @CacheEvict 自动处理
        
        // 3. 发送 MQ 消息 (异步通知其他服务)
        mqSender.send("activity.exchange", "activity.created", Map.of(
            "activityId", activity.getActivityId(),
            "activityName", activity.getActivityName()
        ));
        
        return activity;
    }
}
```

**监听器处理** (ActivityMessageListener.java):

```java
@Component
public class ActivityMessageListener {

    @RabbitListener(queues = "activity.created.queue")
    public void handleActivityCreated(@Payload Object activity, 
                                       Channel channel, 
                                       long deliveryTag) {
        try {
            // 处理活动创建后的逻辑
            // 例如：发送通知、更新缓存、触发 AI 学习
            
            // 手动 ACK
            channel.basicAck(deliveryTag, false);
        } catch (Exception e) {
            // 处理失败，拒绝消息
            channel.basicNack(deliveryTag, false, false);
        }
    }
}
```

**可靠性保证**:

| 环节 | 配置 | 作用 |
|------|------|------|
| **生产者** | confirmCallback | 消息到达交换机确认 |
| **交换机→队列** | 队列持久化 | 服务器重启不丢失 |
| **消费者** | manual-ack | 处理成功后再确认 |
| **失败处理** | dead-letter | 失败消息进入死信队列 |

## 7.8 接口幂等性

### ❓ 面试官问：如何保证接口的幂等性？

### ✅ 参考回答

**幂等性定义**: 同一操作多次执行结果相同。

**常见场景**:

| 场景 | 幂等需求 | 解决方案 |
|------|---------|----------|
| 表单重复提交 | 高 | Token 机制 |
| 支付回调 | 极高 | 唯一订单号 |
| 消息重试 | 高 | 消息去重表 |
| 批量处理 | 中 | 分批 + 断点续传 |

**解决方案对比**:

```
方案 1: Token 机制 (防重复提交)
  1. 前端获取 Token
  2. 提交时携带 Token
  3. 后端验证并删除 Token
  4. Token 不存在 → 拒绝请求

方案 2: 唯一索引 (数据库层面)
  CREATE UNIQUE INDEX idx_order_no ON orders(order_no);
  重复插入 → 主键冲突异常

方案 3: 乐观锁 (版本号机制)
  UPDATE table SET version = version + 1, field = value 
  WHERE id = 1 AND version = old_version;
  影响行数 = 0 → 版本号不一致，拒绝
```

### 📁 本项目中的实现

**Token 机制防重复提交**:

```java
@Service
public class IdempotentService {
    
    @Autowired
    private RedisTemplate<String, Object> redisTemplate;
    
    /**
     * 获取幂等 Token
     */
    public String getIdempotentToken(String userId) {
        String token = UUID.randomUUID().toString();
        redisTemplate.opsForValue().set(
            "idempotent:token:" + userId + ":" + token,
            "1",
            5, TimeUnit.MINUTES
        );
        return token;
    }
    
    /**
     * 验证并删除 Token
     */
    public boolean verifyToken(String userId, String token) {
        String key = "idempotent:token:" + userId + ":" + token;
        Boolean exists = redisTemplate.hasKey(key);
        if (Boolean.TRUE.equals(exists)) {
            redisTemplate.delete(key);
            return true;
        }
        return false;
    }
}
```

**Controller 使用**:

```java
@PostMapping("/activity")
public ApiResponse<Activity> createActivity(
        @RequestHeader("X-Idempotent-Token") String token,
        @RequestBody Activity activity) {
    
    // 验证幂等 Token
    Integer userId = getUserIdFromToken();
    if (!idempotentService.verifyToken(userId, token)) {
        return ApiResponse.error("重复提交", 400);
    }
    
    Activity created = activityService.createActivity(activity);
    return ApiResponse.success(created);
}
```
